# PhishGuard — Plain English Explanation System

## Overview
Every analysed email receives a plain-English explanation describing WHY it was flagged. Generated by the Claude API with a deterministic rule-based fallback if the API is unavailable.

## Pipeline Position
The `generate_explanation` task runs as step 4 of 5 in the analysis chain:

```
parse → extract_features → classify → generate_explanation → apply_outcome
```

## Claude API (Primary)
- Model: `claude-sonnet-4-20250514`
- `max_tokens`: 150
- Timeout: 5.0 seconds
- System prompt: instructs Claude to explain in 2–3 plain English sentences why the email is suspicious
- Input: `top_features` list, sender, subject
- Output: 1–3 sentence plain-English explanation

## RULE_TEXT_TEMPLATES (Fallback)
If the Claude API fails (timeout, rate limit, API error), the system falls back to one of 8 pre-written templates based on the top-scoring NLP feature:

| Feature Key | Explanation Focus |
|---|---|
| `urgency_language` | Pressure tactics and time manipulation |
| `credential_request` | Explicit requests for passwords/personal data |
| `link_mismatch` | Deceptive URLs showing one domain, going to another |
| `impersonation_language` | Brand/organisation spoofing |
| `auth_failure` | SPF/DKIM/DMARC authentication failures |
| `known_bad_url` | URLs in public phishing threat databases |
| `grammar_quality` | Spelling and grammatical errors |
| `default` | General multi-signal phishing characteristics |

## Frontend Display
- **Location**: Email detail view, inside MODEL EXPLANATION panel
- **Position**: Above the feature signals list
- **Style**: Left-bordered card, colour matches risk score
  - Red border (`#ef4444`): score ≥ 80 (phishing)
  - Orange border (`#f97316`): score ≥ 60
  - Amber border (`#f59e0b`): score ≥ 30 (suspicious)
  - Green border (`#16a34a`): score < 30 (safe)
- **Fallback text**: If `explanation` is null, shows italic grey "Explanation not yet available" message

## Database Storage
- **Table**: `analysis_results`
- **Column**: `explanation TEXT` (nullable)
- **Set by**: `generate_explanation` Celery task
- **Always populated**: even on Claude API failure (template fallback after `max_retries=2`)

## API Response
`GET /emails/{id}` includes `explanation` in the `EmailDetail` schema (`Optional[str]`).

## Backfilling Existing Emails
For emails analysed before this feature with `NULL` explanation:

```bash
docker compose exec api python -c "
from app.core.database import get_sync_engine
from sqlalchemy.orm import Session
from app.models.analysis_result import AnalysisResult
from app.services.claude_service import RULE_TEXT_TEMPLATES

with Session(get_sync_engine()) as db:
    results = db.query(AnalysisResult).filter(AnalysisResult.explanation == None).all()
    print(f'Found {len(results)} emails without explanation')
    for r in results:
        top_features = r.top_features or []
        top_name = top_features[0].get('name', 'default') if top_features else 'default'
        r.explanation = RULE_TEXT_TEMPLATES.get(top_name, RULE_TEXT_TEMPLATES['default'])
        print(f'Fixed email_id={r.email_id} with template={top_name}')
    db.commit()
    print('Done')
"
```

## Railway Deployment
- Requires: `ANTHROPIC_API_KEY` environment variable set in Railway
- Without it: All explanations use `RULE_TEXT_TEMPLATES` (still fully functional)
- With it: Claude generates contextual plain-English explanations per email
